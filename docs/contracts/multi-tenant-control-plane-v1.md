# Multi-Tenant Control Plane Contract v1

Date: 2026-06-29
Status: draft contract for implementation
Design source: `docs/superpowers/specs/2026-06-29-multi-tenant-control-plane-design.md`

## Purpose

This contract defines the boundary between the Next.js control plane, the host provisioner, and workspace containers.

The Incus REST API is the provisioner's implementation detail. The web app must not call raw Incus endpoints or hold broad Incus credentials. The web app calls product-level commands such as `CreateWorkspace` and `RunSetup`; the provisioner translates those commands into constrained Incus/ZFS operations.

## Versioning

Contract version: `mtcp.v1`

Breaking changes require a new version. Additive fields are allowed when they are optional and consumers ignore unknown fields.

Every request includes:

```ts
type ContractVersion = "mtcp.v1"
```

## ID Types

IDs are opaque strings. Recommended generated format is UUIDv7, but callers must not parse IDs.

```ts
type UserId = string
type OrgId = string
type WorkspaceId = string
type OperationId = string
type SnapshotId = string
type ResourceProfileId = string
```

## Actor Context

Every control-plane action is evaluated against an authenticated actor.

```ts
type ActorContext = {
  userId: UserId
  oidcSubject: string
  email: string
  displayName?: string
  requestId: string
  ipAddress?: string
  userAgent?: string
}
```

The provisioner does not trust actor-provided authorization claims. The control plane resolves authorization before sending commands, and the provisioner validates workspace ownership/project/container metadata before mutating host state.

## Roles And Permissions

Workspace roles:

```ts
type WorkspaceRole = "owner" | "admin" | "collaborator"
```

Permissions:

| Action | owner | admin | collaborator |
|---|---:|---:|---:|
| View workspace | yes | yes | yes |
| Open terminal | yes | yes | yes |
| Run setup | yes | yes | no |
| Start/stop/restart | yes | yes | no |
| Create snapshot | yes | yes | no |
| Fork workspace | yes | yes | yes |
| Share workspace | yes | yes | no |
| Delete workspace | yes | no | no |
| Transfer ownership | yes | no | no |

Sharing grants shell-level access to the same live container and same Linux user. The UI must show the same-user shared-container warning before creating a grant.

## Workspace State Machine

Allowed transitions:

| From | To | Cause |
|---|---|---|
| creating | stopped | clone prepared but not started |
| creating | running | clone prepared and auto-started |
| creating | failed | create failed |
| stopped | starting | start requested |
| starting | running | start succeeded |
| starting | failed | start failed |
| running | stopping | stop requested |
| stopping | stopped | stop succeeded |
| stopping | failed | stop failed |
| running | restarting | restart requested |
| restarting | running | restart succeeded |
| restarting | failed | restart failed |
| running | setting_up | setup requested |
| stopped | setting_up | setup requested with auto-start |
| setting_up | running | setup succeeded and workspace is running |
| setting_up | failed | setup failed |
| running | degraded | health check failed |
| stopped | degraded | metadata or host state mismatch |
| degraded | running | health recovered |
| degraded | stopped | stopped cleanly |
| stopped | deleting | delete requested |
| running | deleting | delete requested with forced stop |
| failed | deleting | delete requested |
| deleting | deleted | delete succeeded |
| deleting | failed | delete failed |

`deleted` is terminal. A deleted workspace id must not be reused.

## Operation Envelope

Long-running provisioner commands return an operation.

```ts
type OperationStatus = "queued" | "running" | "succeeded" | "failed" | "cancelled"

type Operation<T = unknown> = {
  id: OperationId
  type: ProvisionerCommandType
  workspaceId?: WorkspaceId
  status: OperationStatus
  progress?: {
    phase: string
    message?: string
    percent?: number
  }
  result?: T
  error?: ContractError
  createdAt: string
  updatedAt: string
}
```

The control plane stores operation records and displays them in the UI. The provisioner may expose streaming logs separately, but log excerpts returned through the contract must be bounded.

Provisioner requirements:

- validate container/project names against workspace metadata
- clone from the configured ZFS-backed template
- apply Incus project/profile/resource limits
- attach network ACL

## Terminal Session Contract

Terminal tokens are short-lived and scoped.

```ts
type TerminalSessionRequest = {
  workspaceId: WorkspaceId
}

type TerminalSession = {
  id: string
  workspaceId: WorkspaceId
  userId: UserId
  role: WorkspaceRole
  expiresAt: string
  connectUrl: string
}
```

Rules:

- User must have `owner`, `admin`, or `collaborator` role.
- Token is scoped to one workspace.
- Token expires quickly and should be single-use when practical.
- Opening a terminal writes an audit event.
- The container receives a display label derived from authenticated identity.
- All collaborators attach to the same live container as the same Linux user.

## Incus Adapter Contract

The provisioner may use the Incus REST API over the local Unix socket first. Remote TLS support can be added later without changing the control-plane contract.

Adapter responsibilities:

- discover Incus API extensions required by the current template/workspace flow
- create and select Incus projects
- clone instances from ZFS-backed template snapshots
- start, stop, restart, and delete instances
- create snapshots
- apply profiles and config limits
- configure disk and network devices
- inspect runtime status
- translate Incus operations into `Operation` records
- map raw Incus errors to `ContractError`

Adapter constraints:

- never accept arbitrary endpoint paths from the web app
- never accept arbitrary Incus config key/value maps from the web app
- verify workspace metadata before mutating an Incus instance
- use deterministic generated names
- keep raw Incus error detail server-side

## Security Requirements

- Next.js request handlers must not hold broad Incus credentials.
- Provisioner transport is local Unix socket for v1.
- Provisioner API is not exposed publicly.
- Workspace sharing is high trust and must show explicit warning copy.
- Same live container means same Linux user, shared shell history, shared decrypted workspace state, shared running processes.
- User secrets are not present in the golden image or host template.
- Raw age key material is never written to audit logs or operation logs.
- Terminal tokens are short-lived and scoped to one workspace.

## Compatibility With Current Prototype

Current prototype services map into the contract as follows:

- `incus-web` container becomes an imported `Workspace`.
- current `/setup/` becomes a temporary implementation of `RunSetup`.
- current identity proxy becomes the first terminal identity propagation implementation.
- current deploy script becomes a source for the first provisioner adapter behavior.

The contract allows these pieces to be migrated incrementally without changing the external product model.

## Contract Acceptance Criteria

- A developer can implement Zod schemas directly from this document.
- A developer can build the provisioner API without exposing raw Incus endpoints.
- Workspace role checks are unambiguous.
- Workspace state transitions are explicit.
- Terminal sharing semantics are explicit.
- Setup secret handling is explicit.
- Incus remains an implementation detail behind the provisioner.
